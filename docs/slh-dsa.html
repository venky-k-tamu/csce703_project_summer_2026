What SLH-DSA Replaces — and Why
Motivation for a conservative, hash-based signature standard

ML-DSA (FIPS 204) is NIST's primary post-quantum signature standard, but its security reduces to Module-SIS, a lattice problem that — like all lattice cryptography — is younger and less battle-tested than the number-theoretic problems it replaces. Lattice cryptanalysis is an active research area, and no one can rule out a future structural attack the way decades of scrutiny have (mostly) ruled one out for well-studied hash functions.

SLH-DSA (FIPS 205), standardized alongside ML-DSA and derived from SPHINCS+, is NIST's insurance policy against that risk. Its security rests only on the collision- and preimage-resistance of SHAKE256 — no algebraic structure, no lattice assumption, nothing beyond "this hash function is hard to invert." That minimalism is deliberate: if Module-SIS were ever broken, SLH-DSA's security would be untouched.

The tradeoff is size and speed: hash-based signing means walking large Merkle trees and re-hashing repeatedly, so SLH-DSA-SHAKE-128s signatures run 7,856 bytes (vs. ML-DSA-65's 3,309) and signing is orders of magnitude slower than verification. NIST's guidance is to reach for SLH-DSA where that conservatism is worth the cost — not to replace ML-DSA everywhere.

Where SLH-DSA Fits
A conservative fallback where verification speed and minimal assumptions matter more than signature size
  • Firmware & code-signing roots of trust — devices in the field for a decade-plus, where a hash-only security argument ages better than a newer lattice assumption.
  • Sign-once, verify-often workloads — SLH-DSA's slow signing is a one-time cost at build time; verification is fast, so it fits update-distribution pipelines well.
  • Diversity hedge alongside ML-DSA — standards bodies and long-lived PKI roots that want two independent security foundations, not just two key sizes of the same assumption.
  • Long-lived trust anchors — root CAs and archival signatures where "the assumption held for 20 years" matters more than shaving bytes off the signature.
  • Constrained verifiers — embedded devices that only ever verify (never sign) can absorb the larger signature and still benefit from cheap, hash-only verification.

WOTS+ (Winternitz OTS+)

Structure

Each WOTS+ key pair consists of LEN = 35 hash chains. Each chain is w = 16 steps long. Every step applies tweakable hash F. The secret key is at step 0; the public key top is at step w−1.

Signing

The XMSS leaf message is encoded as LEN1 = 32 base-16 digits plus LEN2 = 3 checksum digits. For chain i with digit di, the signature element is chain[di] — apply F exactly di times from SK.

Verification

Apply F exactly w − 1 − di more times to each signature chain element. The result must equal the stored PK chain top.

Demo uses w = 8, LEN = 8 (illustrative). Actual parameters: w = 16, LEN = 35.

XMSS — eXtended Merkle Signature Scheme

Structure

An XMSS tree of height h′ = 9 has 29 = 512 leaves. Each leaf is a WOTS+ public key. Internal nodes are computed by applying tweakable hash H to the concatenation of the left and right children.

Signing

To sign with leaf index idx: produce a WOTS+ signature on the message, then include the authentication path — the h′ sibling nodes needed to reconstruct the root from the signed leaf.

Verification

Recover the WOTS+ PK from the signature. Walk up the tree using the auth path, hashing at each level. The final value must equal the known XMSS root.

Demo uses h′ = 3 (8 leaves). Actual: h′ = 9 (512 leaves).

FORS — Forest of Random Subsets
Demo uses K = 4, A = 3 (8 leaves per tree) — actual values depend on parameter set

FORS

Structure

FORS consists of K = 14 independent binary trees, each of height A = 12 with 212 = 4096 leaves. Leaves are secret key values derived from SK.seed via PRF.

Signing

The FORS message digest md is split into K A-bit indices. For tree i with index idxi: include the secret leaf value and the A-node auth path to the root.

Public Key Recovery

Hash each secret leaf via F to get the tree leaf hash, walk up using the auth path to get each tree root, then compress all K roots with TK into the single n-byte FORS PK.

Demo uses K = 4, A = 3 (8 leaves). Actual: K = 14, A = 12 (4096 leaves).

Hypertree — D Stacked XMSS Trees
Demo uses D = 4, h′ = 2 (4 leaves each) — actual values depend on parameter set

Hypertree (HT)

Structure

The hypertree stacks D = 7 layers of XMSS trees. Each tree has height h′ = 9. Total height: h = D × h′ = 63. The Layer (D−1) root is PK.root.

Signing

Layer 0 signs the FORS PK using the XMSS tree at idx_tree and leaf idx_leaf. Each higher layer signs the root of the layer below. The signing leaf for layer j is the low h′ bits of the Layer (j−1) tree address.

Verification

Starting from the FORS PK, verify Layer 0's XMSS signature to get its root, treat that root as the message for Layer 1, and repeat up to Layer D−1. The final root must equal PK.root.

Demo uses D = 4, h′ = 2 (4 leaves each). Actual: D = 7, h′ = 9.
